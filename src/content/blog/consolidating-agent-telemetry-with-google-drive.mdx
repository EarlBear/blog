We already had per-machine agent telemetry: each laptop turns its Claude Code transcripts into
derived metrics and pushes them straight to our shared database. It works. But it asks every
machine to hold a database service-role key, it moves *no* transcript content at all — so
there's no shared corpus to render or replay from — and it can't tell you *which* teammate
produced a session. We wanted several machines to pool their telemetry into one place, with
**no server to run**, and we wanted to know who did what.

The obvious idea is the one we spent an earlier design explicitly ruling out: copy the
transcripts somewhere central. We won't. Raw transcripts contain file contents, full prompts,
and pasted secrets, and the one rule we don't break is that raw content never leaves the
machine that wrote it. So the interesting question is the narrow one: *if raw can't move, what
can — and is a shared folder a safe way to move it?*

## The road we didn't take (again)

The tempting transport is `rsync` of the raw files to a central host. It's the standard secure
delta transfer, and it's exactly wrong for us: it moves raw transcripts off the laptop. That
option was rejected on security in our sync design and it stays rejected. Nothing about "a
shared Google Drive folder" changes the calculus for *raw* files.

What changes is *what we put in the folder*. If the only things that ever reach Drive are
already scrubbed and independently verified clean, then the folder is just a courier for
derived, redacted artifacts — the same category of thing we already send to the database. The
invariant holds: raw never leaves the edge. What we relaxed, and we want to be honest that we
relaxed it, is a *stronger* line we used to hold — "we move no transcript bodies at all." Now
we move transcript bodies, but only *scrubbed* ones, and only through a gate described below.

## What actually moves

Each machine writes two files per session into a Google-Drive-for-desktop folder, partitioned
by who and where — `abacus/<you>:<your-machine>/projects/<repo>/`:

- `derived.json` — the metrics and bounded redacted excerpts, exactly as before.
- `events.ndjson` — an append-log of the session's *events of interest*: each tool call,
  subagent run, and skill use, plus a metrics snapshot. One JSON object per line, time-ordered,
  and each event stamped with the running cost and counts as of that point — so you can ask "what
  had this session spent by the tenth tool call?" and the log answers directly.

"Append-log" is a claim worth backing up, because it's easy to *say* while actually rewriting the
whole file each sync. Ours is the real thing: when a session grows, we compare the new events to
what's already on disk and append only the new lines — the earlier bytes don't move. (A `/rewind`
that rewrote history is the exception: there the old lines are no longer a prefix of the new ones,
so we recompute the whole log rather than append onto stale cumulatives. Correctness first.) That
append property is also what lets the reader be lazy: each session's marker carries a hash of its
derived record, so a refresh skips every session that hasn't changed and rebuilds the dashboard's
aggregates only when something actually did.

Drive syncs those files to every other machine. Then any one machine reads the folder and loads
both artifacts into the database: the session metrics into the tables the dashboard already
reads, and the event log into its own table so the per-event cost curve becomes queryable. The
dashboard grew a "cost over the session timeline" panel straight from it — the running-cost data
we stamped on each event is now a line you can look at, not just a number in a file. It's
"manually refreshed as needed" — a command you run, not a service you operate. No servers were
added to make this work.

## The artifact we almost shipped, and why we didn't

Our first instinct was to ship a scrubbed copy of the *whole transcript* — every message, every
tool result, with each string value run through a redactor. It's the obvious "corpus for replay."
It's also where the design nearly went wrong, and the fix is the most useful thing in this post.

Think about the two guarantees. The derived record is protected *structurally*: we build it from
an allowlist of safe fields, so raw content is never copied into it in the first place — an
address or a password that we never named simply isn't there. A full scrubbed transcript is the
opposite shape: we keep everything, including tool results that may contain file contents and
pasted personal data, and lean on a *regex* pass to find what's sensitive. Regex is great at
things with a shape — API keys, card numbers — and blind to things without one, like a name or a
free-form address. Shipping the whole transcript would have quietly made that regex the
*load-bearing* defense against arbitrary personal data. The weakest control, carrying the most
weight.

So we changed the artifact instead of piling more redaction on top of it. The event log is built
from the *same allowlist* the derived record uses: for each tool call we emit the tool name and a
coarse, already-redacted summary — never the raw command or its output. It's a serialization of
fields we already deemed safe, at event granularity. That restores the structural guarantee — no
raw body is ever copied — and demotes the regex layer back to what it's good at: a light backstop
on the two short excerpts, not the primary shield. We kept the full-transcript scrubber for the
one place it's earned: rendering a *specific* session a human has explicitly cleared for
publishing. It's just no longer what syncs to everyone's laptop.

(The event log is plain NDJSON, not gzipped — on purpose. Gzip can't be appended to, so it would
force a full rewrite on every sync and defeat the byte-offset delta model; and an opaque `.gz`
isn't something you can `grep` in the Drive folder. The files are kilobytes; the compression win
isn't worth either cost.)

## The gate in front of Drive

Even with a safer artifact, Drive is a new way for data to leave the machine, so we still gate it
hard. Every egress path we already had
runs a two-detector check — our own redactor plus [gitleaks](https://github.com/gitleaks/gitleaks)
— but treats gitleaks as *best-effort*: if it isn't installed, the check still passes on our own
redaction. For the Drive boundary that posture isn't good enough, because Drive is a new way for
data to leave the machine — so even with the safer event-log artifact we hold the line. The Drive
gate is **fail-closed and requires gitleaks to actually be present and pass**: the self-test runs,
then gitleaks must be installed, then it must find zero secrets in *both* files — and only then
does anything get copied to the folder. No gitleaks, no write. We proved this the way you'd want:
we planted a real token in a transcript and confirmed it never reaches the folder, and we
confirmed that hiding gitleaks makes the write refuse rather than fall back.

## The shape of the flow

Two properties fall out of that shape. The raw file is read-only throughout — we never redact in
place or hand-edit a transcript, and to make that a rule rather than a hope we deny-list both the
raw directory and the Drive output in the editor's own permissions, so the tooling *can't* modify
them. And the completion marker is written *last*, so a folder that Drive is still syncing up in
the background is distinguishable from a finished one — the reader only picks up sessions that are
fully there. Loading is idempotent (upsert by session id), so re-running the refresh never
double-counts.

Along the way we finally get attribution: identity is just `whoami:hostname`, derived with no
configuration, stamped onto each record and carried into the dashboard's metrics — so the numbers
can be read per teammate. A machine rename would split one person across two ids; we wrote that
caveat down rather than papering over it, since the session-id key still keeps the rows from
duplicating.

## When this would be the wrong call

If Drive sync turned out to be unreliable in practice — conflict files, latency that makes the
corpus perpetually half-synced — the fallback is clean: drop back to writing only the derived
record to Drive, or drop Drive entirely and keep the direct database push. And the reverse is
also on the table: if we ever genuinely needed verbatim cross-machine replay, we'd bring the full
scrubbed transcript back — but only for sessions a human has cleared for publishing, with clear
eyes that the regex layer becomes load-bearing again there. The attribution work survives either
way. That's the test we hold every design to — not whether it's permanent, but whether we know
exactly what would make us change our minds.
