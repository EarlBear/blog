## The rule

**Every dashboard mart is gated to a logged-in `@earlbear.com` identity.** All five rollup tables —
`mart_workflow_metrics`, `mart_funnel_daily`, `mart_recent_sessions`, `mart_event_cost`,
`mart_agent_reporting` — carry the same RLS policy: `to authenticated using (auth.jwt()->>'email'
like '%@earlbear.com')`, with anon revoked. The publishable key on its own reads **nothing** from
any mart. Proven live: publishable key alone → 401; an `@earlbear.com` token → 200; a non-earlbear
token → filtered to `[]`.

This page is the *why* behind that rule — so nobody later "simplifies" a mart back to anon-read
without knowing what it costs.

## Why the database needs its own gate

The instinct is: the whole internal site is behind Cloudflare Access — Google SSO, `@earlbear.com`
only, the page is unreachable otherwise — so isn't the data already protected? No. **Cloudflare
Access gates the *page load*, not the API calls the page makes afterward.**

The publishable key is baked into the browser bundle. Any logged-in user can open devtools, copy it,
and replay it with `curl` from anywhere — outside CF Access entirely. So the real question RLS has to
answer isn't "did this person load the page," it's "does this specific request carry a valid
`@earlbear.com` identity." A `using(true)` policy can't tell the difference: it hands the rows to
anyone holding the key. An identity-gated policy rejects the copied key, because a bare key carries
no token.

The gating mechanism itself — a Pages Function verifies the CF Access login and mints a short-lived
`@earlbear.com` Supabase JWT the browser carries on every request — is the same one the scan fleet
uses. It's built and explained in [who are you, to Supabase](/blog/who-are-you-to-supabase); this
page is just about extending it to cover the marts.

## Why we could afford to gate them (the two-build split)

Identity-gating *looks* expensive for a read-only rollup, and for a while that's why the marts were
left anon-readable. The cost was one specific thing: **the snapshot builder can't mint a token.** It
runs at build time, with no logged-in user, so if the marts require an `@earlbear.com` JWT the
builder gets a 403 and the offline/demo snapshot goes empty.

The two-build split dissolves that cost by giving the two readers different keys:

- The **static** build reads the marts at build time with an **RLS-exempt secret key** and ships
  pure keyless JSON — the demo/offline/public-safe artifact. It never needs anon-read.
- The **live** build always reads Supabase in the browser with a **minted `@earlbear.com` token**.

With the split, neither reader relies on an anon-read policy — so gating every mart costs nothing.
Migration `011_marts_identity_prod.sql` drops the `using(true)` policies on all five marts, adds the
`@earlbear.com` `to authenticated` policy, and revokes anon.

For the record, this reversed an earlier call. The original recommendation was to keep `using(true)`
(matching the marts' pre-split state and keeping the snapshot builder working); the split is what
made identity-gating the free, strictly-better option.

## How we found the gap

The gating work started as a bug. Two panels — cost over the session timeline, and agent reporting —
were reading empty on production. They read from `mart_event_cost` and `mart_agent_reporting`, built
by migrations 006/007 — which had been applied to our local database but **never to production
Supabase**. On prod the tables simply didn't exist; the REST API returned 404 (`PGRST205`). The
panels didn't crash — the site falls back to a committed JSON snapshot — so the cost panel showed
stale baked-in data and the agent-reporting panel showed nothing. Quietly wrong, not loudly broken.

Chasing it is what surfaced the whole marts-access question — and produced the drift guard below, so
the *class* of bug can't recur: it reads the frontend's mart list (from `snapshot-data.mjs` /
`useSnapshot.ts`) and asserts each mart exists on prod with a policy. A mart wired into the frontend
but missing from prod now fails a red check instead of silently reading empty.

## Things to watch out for

If you add a mart, touch RLS on one, or debug an empty panel, these are the traps that bit us:

- **404 is not 403.** An empty panel that's a *missing table* (404 / `PGRST205`) looks identical to
  one that's an *RLS denial* (403) until you check the status code. They have opposite fixes —
  create the table vs. fix the policy. Read the code before touching the policy.
- **Empty panels fail silently.** The dashboard falls back to the committed JSON snapshot, so a
  broken live read shows *stale* or *blank* data, never an error. "The panel looks fine" is not
  evidence the live path works — check it under `?live=1`.
- **The Supabase advisor can't see absent tables.** `get_advisors` only lints tables that *exist*,
  so it structurally cannot flag a mart that was never created on prod. A clean advisor report is
  not "every mart is present and policed." The drift guard is what checks presence; the advisor
  checks posture.
- **A new mart must reach prod, not just local + code.** `make db-up` applies migrations locally;
  production is a separate `supabase db push`. Code + local passing tells you nothing about prod.
  The drift guard is there precisely to catch the gap — run it.
- **Don't reintroduce `using(true)`.** It's tempting when a build-time or test reader hits a 403.
  The fix is to give that reader the right key (secret key for build-time, minted token for the
  live app), *not* to open the mart. Reopening it silently restores the leaked-key replay this whole
  page exists to prevent.
- **The static build needs the secret key, the live app needs a token.** If the snapshot goes empty,
  the build-time reader is missing the RLS-exempt secret key; if the live panel 403s, the browser
  isn't carrying a minted `@earlbear.com` token. Same symptom, different key.

The full trust model and the per-key blast radius live in [the key catalog](/blog/every-key-in-the-system)
(whose marts row now reads "identity-gated").
