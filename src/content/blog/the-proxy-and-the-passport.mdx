Every one of our browser apps has the same problem. It's a static site — HTML and JavaScript
on a CDN, no backend of its own. It talks to Supabase directly. And Row-Level Security, the thing
that decides who can read and write what, needs to know *who the caller is*. Cloudflare Access
already logged them in with Google SSO, so we know it's an `@earlbear.com` person — but Supabase
never saw that login. RLS, left alone, can only say "anyone holding the public anon key," which is
everyone.

We solved this twice, in two different parts of the org, and ended up with two genuinely different
architectures. Not by accident, and not because one is wrong. This post is the compare-and-contrast:
what each does, where they're the same, where they diverge, and — the useful part — which one we
now reach for **by default** (the minted token) and the one specific reason you'd still pick the
other. (The token-exchange side has its own deep-dive in
[Who are you, to Supabase?](/blog/who-are-you-to-supabase/); this post is the *comparison* that
post's "why not just use a proxy?" question was pointing at.)

## The same starting point, the same first move

Both approaches begin identically, and it's worth naming the shared part because it's most of the
security:

- The app is behind **Cloudflare Access**. You can't load the page without a valid Google SSO
  session, and Access sets a signed `CF_Authorization` cookie.
- A **Cloudflare Pages Function** — server-side code that runs at the edge, the one piece of
  "backend" a static site is allowed — **verifies** that Access token against Cloudflare's public
  keys and extracts the verified email. This is the trust anchor in both designs. The email is
  never taken from anything the browser could forge; it comes out of a cryptographically verified
  Access JWT.

From here the two diverge on one question: **once the Function knows who you are, how does that
identity reach Postgres?**

## Approach A — the proxy: relay every call with the service key

The first approach (our `earlbear-sites` apps) keeps the browser *away* from Supabase entirely. The
Pages Function isn't just a verifier — it's a **proxy**. The browser sends its database request to
`/api/proxy`; the Function verifies the Access identity, then makes the real Supabase call itself
using the **service key** (which bypasses RLS), forwarding the verified email as an `x-user-email`
header. A small RLS helper function reads that header to know the caller.

the proxy is the sole writer of that header and the browser can't reach Supabase to set it itself.

## Approach B — the passport: mint a token the browser carries

The second approach (our `gtm` dashboard and this blog) does the opposite. The Pages Function
doesn't relay anything — it **mints a short-lived Supabase JWT** (a signed token, ~30 minutes,
carrying the email) and hands it *back to the browser*. The browser then talks to Supabase
**directly**, presenting that token on every call like a passport at a border. RLS reads the email
straight out of the verified token (`auth.jwt()->>'email'`).

The defining trait: **the browser holds a verifiable identity and talks to Supabase directly.**
There's no relay. The Function runs once per session to mint the token, then gets out of the way.
The security boundary is the **signing key** (an EC private key held only in the Function's
environment) — Supabase verifies the token against the matching public key it serves. The browser
never sees the signing key, only the finished 30-minute token.

## Where they're the same

It's easy to overstate the difference, so first the shared ground:

- **Same trust anchor.** Both verify a Cloudflare Access JWT server-side and take the email only
  from that verified payload. Neither trusts a value the browser could set. (A) forwards it as a
  header *after* verifying; (B) signs it into a token *after* verifying.
- **Same "no backend server."** Both live inside a single Cloudflare Pages Function — no
  always-on server, no user table, no session store.
- **Same enforcement point.** RLS in Postgres is the actual gate in both. The Function establishes
  *who you are*; RLS decides *what you can do*. Neither pushes authorization into app code.
- **Same failure mode when unauthenticated.** No Access session → the Function refuses → RLS sees
  no valid identity → the app reads nothing. Both fail closed.

The disagreement is narrow and specific: **does the browser hold a credential and call Supabase
directly, or does it stay keyless and call through a relay?** Everything else follows from that.

**Realtime is the fact that decides most of it.** Supabase Realtime is a WebSocket — a persistent,
direct connection that streams row changes. A REST proxy can relay a request and a response, but it
can't be a live subscription; there's nothing to relay, because the whole point is a socket the
client holds open. So **the proxy cannot do Realtime.** If a feature needs live updates — our fleet
page updating as an agent scans, comments appearing without a refresh — it needs the browser to hold
a direct client, which means it needs a token to authenticate that client. That's approach B, and
it's *why* gtm and the blog chose it.

The proof is in our own code. Even in the proxy-based repo, the one feature that wanted live updates
— a gallery annotations subscription — **bypasses the proxy and opens a direct Realtime channel with
the anon key**, because it has no other option. The proxy handles its reads and writes; Realtime had
to go around it. That seam is the whole trade-off in one file: the moment you want a live socket, the
relay can't carry you.

Realtime is the sharpest edge, but it generalizes into a second point that matters just as much for
us:

- **The whole Supabase surface, not just what the proxy relays.** Realtime is one case of a broader
  truth: the token approach lets the browser use *everything* Supabase offers — REST, Realtime,
  Storage, RPC (`supabase.rpc(...)`), edge functions, embeddings — because it holds a real client.
  The proxy can only expose what we've taught it to relay. Ours forwards REST; the day we want
  Storage uploads or an RPC call, that's *more proxy code to write and maintain*, per capability. B
  gets those for free; A caps you at the relay's feature set. For a stack that reaches for more of
  Supabase over time, that ceiling is the point that compounds.

The remaining rows are real but softer:

- **Per-call Function hop = a latency and rate ceiling.** Every proxied call is an extra network
  hop through *our* Function — added latency, and a Function invocation for every single database
  call. A direct token call goes straight to Supabase with no middle hop and no Function-tier limit
  in the path. At any real call volume, routing *everything* through Functions is a throughput
  ceiling B simply doesn't have.
- **Key in the browser.** The proxy ships *zero* database key to the browser — the service key never
  leaves the Function. The token approach ships a *publishable* key plus the minted token; the
  publishable key is public-by-design and reads nothing on its own (RLS denies it without a token),
  so this is "partial," not a real leak — but it's still one more thing in the bundle.
- **Nothing to sign.** The proxy has no signing key to import, rotate, or protect. The token
  approach's entire security rests on an EC private key living in the Function env — powerful, and a
  real operational responsibility. Fewer moving parts is a genuine point for the proxy.
- **Central choke point.** Because every proxied call passes through one Function, it's a natural
  place to audit, rate-limit, or shape traffic. The direct-token model has no such single point —
  the browser talks to Supabase without us in the middle. (This is the flip side of the hop cost:
  the same relay that adds latency is also the one place you could throttle abuse.)
- **Portability — A's one strategic edge.** Because the proxy is the only thing that talks to
  Supabase, it's a *swappable seam*. If we ever moved off Supabase to another Postgres host or a
  different backend, only the proxy's internals would change — the apps calling `/api/proxy` wouldn't
  notice. The token approach binds the browser to Supabase's client and its JWT/JWKS model, so a
  migration touches every app. This is the one axis where A genuinely wins, and it's a real
  consideration — just not one that outweighs the rest for how we actually build.

## Our default: mint the token (B)

Weighing those rows against how we actually build, **B is our default.** Two forces decide it, and
neither is Realtime specifically:

1. **We don't want to cap ourselves at what the proxy relays.** We reach for more of Supabase over
   time — Realtime for live pages, RPC for server-side logic, Storage, embeddings. A keeps us at
   whatever we've hand-relayed; B gives us the whole surface with no per-capability proxy code. We'd
   rather not build and maintain a relay that always trails Supabase's feature set.
2. **We don't want a Function invocation on the path of every database call.** Routing all reads and
   writes through our own Functions adds a hop and a rate ceiling to *everything*. The token model
   goes straight to Supabase; the only Function call is the once-a-session mint. At volume, that
   difference is real.

The price is honest and we accept it: a signing key to hold as a first-class secret, and a
publishable key in the bundle (inert without a token). Those are bounded, one-time operational costs
— not a tax on every call.

**The one reason to still reach for A** is portability. The proxy is a swappable seam: if we ever
left Supabase, only the proxy changes and the apps don't notice, whereas B binds each app to
Supabase's client and JWT model. That's a genuine strategic hedge — but it's a bet against a
migration we're not planning, paid for with a feature ceiling and a per-call hop we'd feel every
day. We'd rather build directly against Supabase now and treat "move off Supabase" as the thing we
solve *if* it happens, not a cost we pre-pay forever.

So: **B by default; reach for A only when you have a specific reason** — a surface you deliberately
want kept keyless-in-browser and behind a single auditable choke point, or one you expect to outlive
Supabase. Absent that reason, mint the token.

## How we implement B — two reusable packages

A default only sticks if it's *easy* to take. So B lives in two small, reusable packages that any
gated site installs — nobody re-derives the token exchange per app. The split into **two** packages is
deliberate, and it mirrors the exchange's own two halves:

- **`@earlbear/auth-server`** — the Function side: verify the CF Access JWT, assert `@earlbear.com`,
  mint the short-lived Supabase JWT. It's **dependency-free** (Web Crypto only) so it runs inside a
  Cloudflare Pages Function with nothing to bundle. A site's whole auth endpoint is three lines:
  ```js
  import { handleAuthTokenRequest } from '@earlbear/auth-server';
  export const onRequestGet = ({ request, env }) => handleAuthTokenRequest(request, env);
  ```
- **`@earlbear/auth-client`** — the browser side: a singleton `supabase-js` client whose `accessToken`
  callback fetches that token, caches it, and refetches before expiry (carried on REST *and* Realtime).
  A site's whole client is one call:
  ```js
  import { createEbSupabaseClient } from '@earlbear/auth-client';
  const eb = createEbSupabaseClient({ url, publishableKey });
  ```

They're **two** packages, not one with two entry points, so the boundaries can't leak: the server half
depends on `supabase-js`, the browser SDK — installing the dependency-free server half in a Function
can never transitively pull that in. It's the same discipline as the architecture the post is about —
keep the runtimes' dependencies from cross-contaminating — applied to our own package layout. The blog
(comments) and the gtm dashboard both consume these two packages; the only per-app code left is *where
the URL and key come from* and the RLS policy itself. There's a companion `wire-supabase-auth` skill
that walks a new site through installing them, so the default is a checklist, not a re-derivation.

## The honest cost of having both

Standardizing on B doesn't erase A — the `earlbear-sites` proxy is shipped and working, and there's
no urgency to convert it. But two architectures for one problem is a real tax: two mental models, two
places a new engineer learns "how auth works here," two failure modes to debug. The resolution isn't
a rewrite; it's a **default**. New surfaces take B unless they have the specific reason above, so the
split stops growing and A becomes the documented exception rather than a coin-flip. (The next step
after this post is to make that default *executable* — a skill that stands up B's auth on a new site
so nobody re-derives it, and A stays a deliberate opt-out.)

## When this decision would change

The token wins today partly because a REST proxy can't carry a WebSocket and can't reach past the
capabilities we relay. If Supabase (or a gateway in front of it) offered a way to proxy an
authenticated Realtime connection *and* transparently pass through RPC/Storage/etc. — verifying
identity at each layer the way the proxy verifies REST — then A's feature ceiling would fall away and
its portability edge would tip the balance back. The other reversal is scale of a different kind: if
we committed to staying backend-agnostic (a real chance we leave Supabase), the swappable-seam
argument for A gets much stronger. Neither is true today, so the default stands: **mint the token.**
