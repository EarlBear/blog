Someone signed out of an internal EarlBear app, clicked back in, and asked a fair
question: *why am I on `cloudflareaccess.com`?* Our staff apps —
`wireframes.earlbear.com`, and now a couple more — sit behind Cloudflare Access
with Google sign-in. The gate works. But a signed-out visitor's first sight was a
bare `cloudflareaccess.com` screen with a cryptic token in the URL. It looks like
someone else's site. For a page that's supposed to say "this is EarlBear, sign in,"
that's the wrong first impression.

The fix seemed obvious: put *our* branded page in front. We had one already —
`login.earlbear.com`. Send unauthenticated visitors there first, let them click a
trustworthy "Sign in," then hand them off to Google. How hard could it be.

This is the story of building that, watching our own proof-of-concept tests take it
apart one assumption at a time, and shipping something much smaller instead. The
short version: the best outcome of the week was deleting most of what we wrote.

## The plan that looked right

Cloudflare Access won't let you host its login on your own domain — the sign-in
page stays on `cloudflareaccess.com`, and you can only brand it a little. So we
couldn't move the page. Instead we'd *intercept before* it: a Cloudflare **Redirect
Rule** at the edge, firing on any unauthenticated request to a gated host, sending
the visitor to `login.earlbear.com` with their original destination attached so we
could return them there after sign-in.

We wrote it down as a design with options and a rule we hold ourselves to: no
option gets chosen until at least two are written out honestly, and any claim the
design leans on gets a proof-of-concept before we commit. That rule is the only
reason this story has a happy ending.

The design leaned on one big claim, and we wrote it in bold: *Cloudflare Access
appends its own query parameters during login, so if our return destination rides
in the query string too, the two collide and ours gets mangled.* From that single
claim the whole elaborate shape followed — an opaque encoded token instead of a
plain URL, a dedicated decode route, the works.

Then we started proving the claims. That's where it fell apart — usefully.

## The proofs that said no

The first surprise was small and mechanical. We wanted the edge rule to base64-encode
the destination so it couldn't collide with anything. Cloudflare's rules language has
an `encode_base64` function, so this seemed fine. It is not fine: the redirect target
rejected it outright — *"the use of function encode_base64 is not allowed"* — and,
separately, *"concat … exceeding the maximum allowed number of calls 1."* The edge
simply cannot build the token we designed. Fine, we moved the encoding into a small
function on the login side and pressed on.[^p2]

The second surprise ended the whole approach. We put the redirect live and clicked
through it as a real signed-out user. Landing on `login.earlbear.com` worked. Clicking
"Sign in" did *nothing* — it bounced straight back to the same page. A loop.

The cause is almost funny in hindsight. Our "Sign in" button pointed at the gated
host, because that's what triggers Cloudflare Access. But the visitor was *still
unauthenticated at that moment* — so our own edge rule caught the click and sent them
right back to the login page. Every path on the gated host was intercepted, including
the one path that was supposed to escape into Google's sign-in. We'd built a door that
locked behind itself.[^p4]

And here's the part that stings: **the original premise was wrong too.** Cloudflare
Access does append its own parameters — we confirmed that.[^p1] But it appends them on
the `cloudflareaccess.com` hop, not on `login.earlbear.com`. Our return destination
never shares a query string with Cloudflare's parameters, because they live on
different hosts. The collision we architected a whole encoded-token scheme to avoid
*could not happen on the leg where we'd have used a plain URL.* The scary claim that
justified the complexity was true in general and irrelevant in particular.

## The setting we'd missed

At this point the honest move was to stop defending the design and go read the docs
we'd skimmed. One search later: Cloudflare Access has a per-application setting called
**"Apply instant authentication."** With a single identity provider — we have exactly
one, Google Workspace — it skips the Access page entirely and sends the user straight
to the provider's sign-in.

We enabled it, deleted the edge rule, and watched. A signed-out visitor now goes:

There is no branded interstitial. There didn't need to be. The visitor effectively
sees their own Google sign-in — the page they trust most — and lands where they were
going. The `cloudflareaccess.com` stop still technically happens, but it's a
sub-second script that forwards to Google before anything renders; we confirmed in the
browser that a genuinely signed-out session (`auth_status: NONE`) goes straight to
Google's account chooser, scoped to `earlbear.com`.[^p10]

One setting. Zero custom code. And it's *more* secure than what we built, precisely
because there's no custom redirect surface to get wrong — and we had already gotten it
wrong twice.

## What we deleted, and what we kept

Gone: the edge Redirect Rule, the script that generated it, a dedicated Cloudflare API
token minted just for it, the login-side decode function, and a whole branch of our
domain validator that checked for a redirect that no longer exists. The design document
survives as a gravestone with the proofs attached, so the next person who has this
idea can read why it doesn't work before spending the week.

Kept: everything the detour taught us to build *around* the redirect. Onboarding a new
gated subdomain is now a single codified path — create the Pages project, attach the
domain, create the Access application over the API, add the sign-in policy, record it
in one manifest, validate. No dashboard clicking. We proved it by standing up two new
gated apps end to end while chasing this. That infrastructure was never the dead end;
only the redirect was.

## The lesson we keep relearning

Twice in one week, a claim we were confident about was refuted by the smallest possible
real test — a rejected API call, a single click that looped. Both times, the thing that
saved us wasn't being smart up front. It was the rule that says *prove the load-bearing
claim before you build on it,* and then actually running the proof instead of admiring
the design.

The reward for that discipline is usually not a clever system. It's permission to
delete one. The best login page we shipped this week is the one we didn't build.

The collision risk is real; it just doesn't occur on `login.earlbear.com`, which is a different host.
