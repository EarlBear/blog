*New here? [Start with the whole system](/blog/start-here-the-whole-system) — what EarlBear is and the stack this runs on.*

This is the source of truth for **every credential in the go-to-market stack** — kept honest by the
`audit-keys` skill, which enumerates keys across the bundle, CF Pages, `.env`, and the provider
consoles and cross-checks them against this page. It answers the questions a principal security
engineer asks of each key: *what it is, how it's acquired, where it lives, whether it's exportable,
who can read it, its blast radius if leaked, and how it rotates/revokes.*

The governing idea: **three environments, and a key's safety is defined by which one holds it.**

## The inventory

Every key, scored on the dimensions that matter. **Exportable** = can the raw value be read back
out. **Blast radius** = what an attacker does with it. The two that deserve the most respect are the
**signing key** (mint tokens as us) and the **secret key** (bypass RLS) — both server-side only.

| # | Key | Type | Acquired | Lives in | Encrypted at rest | Exportable | Who reads it | Rotation / revocation | Blast radius if leaked |
| --- | --- | --- | --- | --- | --- | --- | --- | --- | --- |
| 1 | **Publishable key** (`sb_publishable_…`, `SUPABASE_ANON_KEY`) | API key | Supabase API-keys tab | `build:live` browser bundle (NOT the `build:static` artifact — that ships keyless) | n/a (public) | yes (it's public) | anyone who loads the gated page | rotate in Supabase; independent of other keys | **Very low.** Reads only what RLS allows — and as of migration 011 **every mart is identity-gated**, so the key alone (no minted `@earlbear.com` JWT) reads **nothing** from any mart (proven: 401). It's inert without the token. |
| 2 | **Signing key** (`SUPABASE_SIGNING_KEY`, EC P-256 private JWK) | asymmetric private key | `supabase gen signing-key` → imported | **CF Pages secret** | yes (CF-managed) | no (never leaves CF env; the browser gets a 30-min token, not the key) | the `/api/auth-token` Function only | rotate = import new + rotate; revoke via Management API. Tokens keep verifying until the key is revoked. | **HIGH.** Sign a JWT as *any* `@earlbear.com` user → pass all identity-gated RLS + Realtime. The one credential that most matters. |
| 3 | **Secret key** (`sb_secret_…`, `SUPABASE_WRITE_KEY`) | API key | Supabase API-keys tab (scoped `earlbear_edge_fns`) | operator env (MCP reporting server) | via operator env | reveal-once in dashboard | the MCP sink process only | rotate: mint new scoped key, swap, revoke old | **HIGH.** Bypasses RLS entirely — read/write every table. 401s in a browser by design. Never in the bundle. |
| 4 | **Cloudflare API token** (`CLOUDFLARE_API_TOKEN`) | API key | Cloudflare dashboard (scoped Pages\:Edit) | `.env` (dotenvx-encrypted) | yes (`encrypted:`) | via dotenvx + `.env.keys` | `make deploy` / operator | rotate in CF dashboard; scoped so blast radius is bounded | **Medium.** Deploy to the gtm Pages project + set its secrets. Scoped to Pages, not account-wide. |
| 5 | **CF Access service token** (`CF_ACCESS_SVC_*`) | credential pair | CF Zero Trust console | `earlbear-domain/.env` (dotenvx) | yes (`encrypted:`) | via dotenvx | origin-probe automation | rotate/revoke in CF Zero Trust | **Medium.** Passes the CF Access gate (as a machine, no email) → reach gated app origins. Can't mint identity tokens. |
| 6 | **dotenvx private key** (`.env.keys`) | symmetric key | `dotenvx encrypt` (first run) | operator laptop + **LastPass** (source of truth) | n/a (it *is* the key) | it's local-only; backed up in LastPass | operator | rotate = re-encrypt all values; back up to LastPass | **HIGH (local).** Decrypts every `encrypted:` value in `.env` (CF token, keys). Never committed; gitignored. |
| 7 | **Supabase PAT** (`SUPABASE_ACCESS_TOKEN`) | OAuth-ish admin token | `supabase.com/dashboard/account/tokens` | **ephemeral** — shell env for a one-time task | no (was temp) | yes | operator, briefly | **single-use → revoked after the signing-key import** | **CRITICAL (while live).** Account-wide admin: create/rotate keys, read config. Which is why it's minted for one task and revoked immediately. |

Two values the Function also reads — `CF_ACCESS_TEAM_DOMAIN` and `CF_ACCESS_AUDIENCE` — are
**identifiers, not secrets** (the team domain and the Access app's AUD tag). They live server-side so
a request can't spoof them, but leaking them grants nothing.

## The honest caveat: `using(true)` and the publishable key

The question that prompted this page: *is a `using(true)` table actually secure?* The straight
answer: **it's protected by "you need the publishable key, which requires having loaded the gated
page once" — which is weaker than it looks.**

<FlowDiagram
  title="Why using(true) is weaker than identity gating"
  shape="branch"
  legend
  desc="Cloudflare Access gates the page load, not subsequent API calls. A logged-in user can extract the publishable key from the bundle and replay it via curl from outside CF Access to read any using(true) table. An identity-gated table rejects that copied key because it carries no token."
  nodes={[
    { id: 'key', label: 'copied key', kind: 'external', detail: 'The publishable key, extracted from the bundle by any @earlbear.com user who loaded the page once.' },
    { id: 'curl', label: 'curl from anywhere', detail: 'Replayed OUTSIDE CF Access — the gate only covers the page load, not a direct REST call with the key.' },
    { id: 'open', label: 'using(true) table', kind: 'store', detail: 'READS — any holder of the key gets the rows. This is the GH/StatiCrypt path (and was the marts, pre-migration-011).' },
    { id: 'gated', label: 'identity-gated table', kind: 'store', detail: 'REJECTS — the copied key carries no @earlbear.com JWT, so RLS returns nothing. This is the scan fleet AND all five marts (post-011).' },
  ]}
  edges={[
    { from: 'key', to: 'curl', label: 'replay' },
    { from: 'curl', to: 'open', label: 'reads ✓' },
    { from: 'curl', to: 'gated', label: 'denied ✗' },
  ]}
/>

So `using(true)` is *tolerable* for **non-sensitive, read-only rollups** where the worst case is
"someone read aggregate numbers" — but "tolerable" is not "right," and **we moved the marts off it.**
Migration `011_marts_identity_prod` drops the `using(true)` policies on all five marts and gives them
the same `@earlbear.com` token-exchange gate as the scan fleet. What made that "not worth it" before
— the build-time snapshot builder can't mint an `@earlbear.com` token, so gated marts would break the
build — dissolved once we split the build in two: a **static** build reads the marts at build time
with the RLS-exempt *secret* key and ships pure keyless JSON, and a **live** build always carries a
minted token. Neither path needs anon-read on a mart, so `using(true)` bought us nothing and cost us
a weaker posture. The lesson generalizes: on this system, treat `using(true)` as a finding on *any*
prod table, not just sensitive ones. (The scan fleet earned its gate first because it also has anon
*writes* — a leaked key could enqueue junk — but the marts have now caught up on reads.)

## What this page is missing (and the audit will chase)

Applying the full principal-engineer question set honestly surfaces where we're thin — these are
open, not solved:

- **Audit logging** (Q16): we don't centrally log *who read which key when*. Provider-side logs
  exist (Supabase/CF), but there's no unified access trail.
- **Rotation cadence** (Q11): most keys have no scheduled rotation — they're rotate-on-suspicion.
  The signing and secret keys especially should have a defined interval.
- **Leak detection** (Q15): gitleaks + the pre-commit hooks cover *commit-time* leakage; there's no
  runtime detection of a key exfiltrated from the bundle and replayed.
- **Incident playbook** (Q17): revocation procedures exist per key (documented in
  `manage-credentials`), but there's no single "key X leaked → do these steps in this order" runbook.

The `audit-keys` skill re-runs the enumeration and this question set on demand, and updates this
catalog when a key is added, moved, or rotated.

## Advisor findings — retrieved, triaged, and CLEARED

Running the Supabase security advisor (mandated after every migration — see the `security-posture`
skill's advisor workflow) surfaced a batch of findings, **all now fixed** (migration
`013_advisor_hardening_prod`, 2026-07-09). Each was checked against real usage first — the readers/
writers use the service-role key (RLS-exempt), so none of the fixes broke a live path:

- **`security_definer_view` (ERROR ×3)** — `v_ga_readiness`, `v_rework_tracker`, `v_cost_summary`
  (read-only aggregate views over the agent-tracker tables) → flipped to `security_invoker = true`.
  Verified the views still return rows. **Fixed.**
- **`rls_policy_always_true` (WARN ×2)** — `instagram_posts` / `instagram_sync_log` had anon *write*
  (`ALL`, `WITH CHECK(true)`) policies → dropped + anon writes revoked (the IG sync runs server-side
  with the service key). **Fixed.**
- **`function_search_path_mutable` (WARN ×4)** — `set_updated_at`, `current_user_email`,
  `insert_agent_run`, `insert_agent_issue_run` → pinned `search_path` (`public` for the
  `SECURITY DEFINER` write functions, `''` for the trivial two). **Fixed.**
- **`public_bucket_allows_listing` (WARN)** — the `instagram-media` bucket's broad listing policy →
  dropped (bucket is `public=true`, so object URLs still serve). **Fixed.**
- **`auth_rls_initplan` (PERF WARN ×5)** — our own mart policies called `auth.jwt()` per row →
  wrapped as `(select auth.jwt())` (one-time InitPlan; gating identical, 15/15 tests still pass).
  **Fixed.**

The security advisor now returns only the intended **INFO `rls_enabled_no_policy`** lints — the
`cc_*`/`agent_*`/`team` internal tables that are RLS-on/no-policy by design (nobody reads via anon).
