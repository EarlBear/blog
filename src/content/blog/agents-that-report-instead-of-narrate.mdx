We asked an agent to review a Shopify storefront the way a first-time shopper would. It did a
genuinely good job: it found that the two products you reach first from the men's collection
were both sold out with the Add-to-Cart button disabled, that the PDP had no reviews to
reassure a new buyer, and that a prominent "up to 2-week shipping delays" banner sat right at
the top of the funnel. All correct. All useful.

And all of it was **prose**, buried in a conversation transcript. To learn what the review
found, you had to read the whole thing. There was no count of issues by severity, no way to
plot the latency it measured, no reconstruction of the path it took to checkout, and — the
part that stung — no way to *watch* the findings arrive while the review was still running. The
agent had done the work; the work was trapped.

So we changed what "reporting" means. Instead of the agent narrating findings into the
transcript, each observation is a **tool call**.

## The reporting is the call

The agent doesn't write "I noticed the PDP has no reviews." It calls a tool:

```
report_issue({
  run_id: "rev-20260708-2100-d032",
  title: "No reviews / star ratings on PDP undercuts purchase confidence",
  severity: "warning",
  category: "pdp",
  page_url: "https://www.allbirds.com/products/mens-tree-runners",
  detail: "Social proof is the top question a first-time buyer has; the PDP shows none.",
})
```

That single call does two jobs at once. It lands in the session transcript as a `tool_use`
block — so the same telemetry pipeline that already mines our Claude Code sessions can roll
every `report_*` call into a table with no new extractor code. And the tool's server writes
the report to disk the instant it's made — so a dashboard can read it *now*, without ever
talking to the agent.

The tools are typed, not one generic `report(payload)`. There's `report_issue`,
`report_positive`, `report_friction`, `report_journey_step`, `report_metric`, and
`report_progress`, bracketed by `start_review` and `end_review`. Typing them was a real
choice with a real cost — six schemas to maintain instead of one — and we took it on purpose.
A typed call is self-describing in the transcript (the block carries `severity`, `category`,
`page_url` as named fields), the model gets per-field validation, and each type maps one-to-one
to a facet on the dashboard. A generic tool would have pushed all of that discipline back into
prose, which is exactly what we were trying to escape.

## What "watch it live" actually requires

The tempting way to stream a live feed is to keep rewriting one JSON file per run — read the
whole thing, append the new report, write it back. It fails the moment a reader shows up while
you're mid-write: it gets a truncated, unparseable document and the page goes blank. And a
polling dashboard shows up *constantly*.

So the on-disk format is append-only. Each run is a `.jsonl` file, one report per line, plus a
small `index.json` (rewritten atomically via temp-file-and-rename) that lists the runs. An
append of a single line is atomic enough that a reader sees each line whole or not at all,
never half. The reader tolerates a torn final line — skip it, it'll be complete on the next
read — and keeps its last good state on any error. That one property, torn-read safety, is
what lets the page poll (or stream) a file that's actively being written without ever flashing
empty.

For the live effect itself we didn't reach for a WebSocket. The traffic only flows one way —
server to browser — so Server-Sent Events over a file watcher is the honest fit: when the MCP
server appends a line, the watcher pushes it, and the report appears on the page in about a
tenth of a second. Polling stays as the fallback for the offline snapshot and for when the
stream drops. Either way the feed *appends* the new report with a small entrance animation
rather than re-rendering the list, so it reads as a stream even when it's really a poll.

## How the pieces fit

Those three scenes are one path through a handful of files. The plugin's MCP server is the only
thing that writes; everything downstream reads the same on-disk contract. The reports directory is
the seam — the plugin never knows the dashboard exists, and the dashboard never imports the plugin.

The transcript path is the second reader: the same tool calls that the server persists also land
in the session JSONL, which the existing telemetry pipeline extracts into `cc_tool_calls` and
migration 007 rolls into `mart_agent_reporting` — so the dashboard can cross-check that the number
of reporting tool calls in the transcript matches the number of reports on disk.

## Attaching evidence

An issue reads better with a picture. So `report_issue` (and positives and frictions) can carry
media: the agent captures a screenshot to a file and passes the path, and the server ingests it
into the run so the report is self-contained. It can also pass a **highlight box** — the exact
`getBoundingClientRect` of the offending element — which the dashboard draws as a rectangle over
the image, plus **text annotations** pinned to points on it. And when a single still can't do the
finding justice — a layout that jumps as it loads, a swatch that flickers when you switch color, a
janky transition — the agent records a **GIF** instead. The server detects the `.gif` and the
dashboard plays it inline. A glitch you can only *see in motion* becomes a report you can watch.

## North star: the report becomes an explorable map

Right now the review is a *stream* — an ordered feed of events, with the JourneyMap giving it a
spine. But the agent is doing something richer than walking a line: as it hops home → collection →
PDP → cart, it's discovering the *shape* of the store. Every navigation is an edge; every page a
node; every issue, positive, and friction a marking on a node. The stream is a flattening of a
**graph** that the agent is implicitly building as it explores.

The next step is to make that graph first-class: a sitemap that *accretes* as the review runs, and
an explorable force-directed view of it — pages as nodes sized by how many findings landed there,
edges as the paths the agent actually took (not the ones a crawler guesses), colored by outcome.
Click a node, see its findings and its screenshot; watch the map grow live as the agent reaches new
pages. The JourneyMap is the happy-path slice through this; the graph is the whole territory,
including the dead ends and the detours the linear stepper can't show.

This isn't a new idea to invent from scratch — two bodies of prior art point the way. On the
**visualization** side, SEO crawlers already render sites as interactive graphs: Screaming Frog ships
a 3D force-directed crawl visualization, Sitebulb draws crawl maps, and open-source crawlers lean on
`vis.js` / D3 force layouts for exactly this node-edge-with-metadata picture. We'd borrow the layout,
not the crawler — our edges come from the agent's real path, which is the interesting difference.

On the **agent-memory** side, the graph is also the agent's *own* working memory of the site, and
recent research treats site memory exactly this way: WebATLAS builds a cognitive map of a site an
agent has explored so later runs navigate by memory instead of re-crawling; state-space navigation
work (e.g. Laser) models web tasks as movement through a graph of page-states; and temporal knowledge
graphs (Graphiti) capture how that structure and its facts change over time. So the same graph serves
two masters — a human's explorable report *and* the agent's reusable memory of the store — which is
why it belongs in the north star and not just the UI backlog.

## What we accepted, and when this is wrong

The honest costs: the live stream only exists under the dev/preview server (the middleware that
serves the reports directory), so a deployed build shows a committed snapshot rather than a live
feed — which is fine, because live-watching is a local act. We hand-rolled the MCP server's
JSON-RPC framing so the plugin installs with zero dependencies and no build step, trading a few
hundred lines of protocol code for no install friction. And we're the first plugin in our
marketplace to bundle an MCP server at all, so we're setting that convention rather than following
one.

This would be the wrong design if reports per run grew large enough that a file-watcher-plus-SSE
stream strained, or if runs became long-lived and highly concurrent — then persistence belongs in a
real store and the stream behind a managed backend. But for an agent reviewing a storefront and
telling you, in structured events you can count and watch, exactly where the path to checkout
breaks, making the report *be* the call is the whole trick.
