The reporting MCP server had one job and one sink: an agent calls `report_issue` (and its
siblings), and the server appends the finding to a per-run JSONL file and rewrites a small
index. Zero dependencies, all local. That's what the live `/reports` page reads.

Then we wanted a *shared* fleet page — a wall of Shopify sites updating live for the whole
team, not just the operator watching their own filesystem. That means the findings need to land
in Supabase too. The question was how to add a second sink without disturbing the first.

## Two choke points, not eight handlers

The server has eight reporting tools, but every one of them funnels through the same two
functions: `appendEvent` (writes one finding) and the run lifecycle (`start_review` /
`end_review`). So the entire second sink is a few lines at those choke points — no handler
touched:

```
function appendEvent(runId, partial) {
  // ... unchanged: append the JSONL line, bump counts ...
  emitToSupabase('scan_events', eventRow(event), { onConflict: 'run_id,seq' })
  emitToSupabase('scan_runs',   runRow(run.meta), { onConflict: 'run_id' })
  return event
}
```

`start_review` additionally upserts the first-class site row and, if the review is fulfilling a
queued request, claims it (`queued → scanning`). `end_review` closes it (`scanning → done`) and
rolls up the **domain-level metrics** onto the site — total issues, positives, frictions, and
the distinct session ids seen across every run of that domain.

## Zero dependencies, still

The server has no HTTP client and we didn't want to add one. Node ships `https`, so the sink is
a plain `https.request` to Supabase's REST endpoint (`/rest/v1/<table>`) with the key in the
headers and `Prefer: resolution=merge-duplicates` — which makes every write an idempotent upsert
on the table's primary key. Retry the same event and you get one row, not two. Still zero
dependencies.

## Best-effort, so the file contract never breaks

The Supabase sink is additive and *best-effort*. Every request has a short timeout, and any
failure — a network blip, a 4xx, Supabase down — is logged to stderr and swallowed. The file
append already happened; the mirror is a bonus. A Supabase hiccup can never break the local file
contract the `/reports` page depends on. And when the env vars aren't set at all, the sink is
simply off: no network, byte-for-byte the original file-only behavior.

## The key stays on the laptop

Writing `scan_runs`/`scan_events` requires a **service-role** key — Row-Level Security blocks
the public anon key from those tables on purpose (the whole point is that the public page can
read and enqueue, but only the trusted agent writes the record of what it found). The server
reads that key from `process.env`, passed through its config. It lives in the operator's shell
environment and nowhere else: not in the frontend build, not in `public/`, not in a commit. The
config we ship references the env var by name; the value is the operator's to supply. One
credential, one machine, one boundary.
