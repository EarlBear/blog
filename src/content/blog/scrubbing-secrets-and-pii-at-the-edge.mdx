Our telemetry pipeline reads raw Claude Code transcripts and turns them into dashboard
metrics. Those transcripts are the worst-case input for a privacy pipeline: full prompts,
pasted API keys, file contents, the occasional customer email. Nothing secret is allowed to
reach the database. So we had to answer a question every team eventually faces — *how do you
actually scrub this stuff?* — and the answer turned out to be a spectrum of techniques, not a
single tool. Here's the tour, ranked roughly from strongest to weakest, with where each one
earns its place.

## Technique 0: don't collect it (structural allowlist)

The strongest scrubber is the one that never sees the secret, because the secret was never
copied in the first place. Instead of taking a raw transcript record and deleting the sensitive
fields, we *construct* a new record from an explicit allowlist of safe fields — token counts,
tool names, timings, two short excerpts. A file body has no field to be copied from; it simply
isn't in the allowlist, so it never enters a record to be scrubbed.

This is the inversion that matters. Deny-listing ("copy everything, then remove the bad parts")
fails open: anything you didn't anticipate rides along. Allow-listing ("copy only the named good
parts") fails closed: anything you didn't anticipate is dropped by default. Every technique below
is a *detection* technique — it looks at data and decides what's sensitive — and detection can be
wrong. The allowlist is the only one that can't leak an unknown unknown, because it isn't
guessing. That's why it's technique zero: it's not on the same axis as the rest.

The catch is that it only works where you control the schema. For the two short excerpts we *do*
keep — the first prompt and a coarse per-tool summary — there's real free text, and now we're in
detection territory. (We used to also sync a scrubbed *full* transcript between machines, which
was a much larger free-text surface; we later replaced it with an allowlist-built event log
precisely to shrink that surface back down — more on that at the end.)

## Technique 1: regex named patterns

Most secrets have a shape. An AWS key is `AKIA` plus sixteen uppercase-and-digits; a GitHub token
is `ghp_` and a base62 run; a JWT is three base64url segments joined by dots. Regex nails these:
it's deterministic, fast, zero-dependency, and precise on structured tokens. Our own redactor is
a list of named patterns applied specific-first, so the label on a redaction is the most specific
match — `[REDACTED:anthropic-key]`, not a generic `[REDACTED:token]`. This is exactly how the
rule engines in the well-known scanners work too: [gitleaks](https://github.com/gitleaks/gitleaks)
ships a large TOML library of credential regexes, and Microsoft's
[Presidio](https://github.com/microsoft/presidio) calls the same idea a "recognizer."

Regex's limit is equally sharp: it only catches what has a shape. A person's name, a mailing
address, a password that looks like a word — these have no regex. Pattern matching also misses
context: `4111 1111 1111 1111` is a credit-card test number in one paragraph and a meaningless
digit string in another, and regex can't tell them apart. For structured identifiers regex is the
right tool; for *unstructured* PII it's a floor, not a ceiling.

## Technique 2: entropy (and what's replacing it)

The gap regex leaves is the *novel* secret — an internal token format you've never seen. The
classic catch-all is Shannon entropy: a long, high-randomness string is probably a key even if no
pattern names it. We run an entropy sweep as the last pass, over runs of 32-plus key-ish
characters, and — this part matters — we *exempt* UUIDs and tool-call ids, because those are
high-entropy identifiers that we need as database keys and that leak nothing. Without that
exemption an entropy filter over-redacts every id in the file.

Entropy is a blunt instrument, and 2026 has been quietly retiring it. Betterleaks, a drop-in
gitleaks successor from gitleaks' original author, replaces the Shannon-entropy signal with a
byte-pair-encoding token-efficiency measure and reports 98.6% recall on the CredData benchmark
against 70.4% for the entropy-based filter it replaces.[^bpe] The insight is that real secrets are
*token-inefficient* — they don't compress the way English or even base64-of-something-meaningful
does — and a tokenizer captures that far better than raw character entropy. If you're building an
entropy pass today, that's the direction to look.

## Technique 3: NER and zero-shot models

For the PII that has meaning but no shape — names, places, organizations — you need a model that
reads context. This is where named-entity recognition comes in, and it's the other half of
Presidio: regex recognizers for structured entities, plus a spaCy NLP model for contextual ones
like person names. The newer generation is *zero-shot*: [GLiNER](https://github.com/urchade/GLiNER)
lets you hand it a list of labels ("first name", "SSN", "credit card") and finds them without
retraining.

The models are genuinely better at what regex can't do — and genuinely worse in ways worth being
honest about. On a multi-domain PII benchmark, GLiNER-base topped the F1 chart around 81%, well
above a plain regex or spaCy baseline.[^gliner] But that average hides a precision problem: on
English phone numbers the same class of model hit about 92% recall against only about 35%
precision[^precision] — it finds nearly every phone number and flags a pile of non-phone-numbers
alongside them. High recall, moderate precision means *over-redaction*: safe for privacy, rough on
readability. Models are also heavier: a hybrid regex-plus-NER pipeline in one 2026 write-up added
roughly 30–40% CPU overhead under sustained load.[^overhead] And the tempting move of bolting regex
onto an NER model to get the best of both mostly doesn't pay: a 2026 cross-domain benchmark found
the gain from adding regex to a good NER model was within noise.[^combo] Reach for NER when you
truly have unstructured PII to catch; don't reach for it expecting a free lunch.

## Technique 4: verification-first secret scanning

The last technique changes the question from "does this look like a secret?" to "is this secret
*real*?" [TruffleHog](https://github.com/trufflesecurity/trufflehog) finds candidates with regex
and entropy like everyone else, then makes a live API call to the provider to check whether the
credential actually works right now. A verified finding is high-confidence by construction — it's
not "matches the GitHub token shape," it's "GitHub accepted this token."

That power comes with cost and coupling: the verification phase makes a scan many times slower than
a pattern-only pass, it needs network access, and it can hit provider rate limits. So the common
production pattern isn't one tool — it's two, matched to where they run: a rule-first scanner like
gitleaks at the edge where speed and offline operation matter, and a verification-first scanner like
TruffleHog on a schedule where confidence matters more than latency.

## How we actually layer it

None of these is sufficient alone, so we don't pick one — we stack them by strength and fail
closed:

Two principles hold the stack together. First, **defense in depth with a second, independent
detector**: our regex is the primary control, and gitleaks — a different rule library written by
different people — is the check on it, so a gap in our patterns has a second chance to be caught.
Running two scanners at the edge is the standard recommendation, not a belt-and-suspenders
indulgence. Second, **fail closed**: a planted-secret fixture runs before every push, and if a
known secret ever survives redaction the pipeline stops instead of shipping. Treating a leak as a
failed unit test — wired into the same gate as the rest of the build — is the difference between
catching it in development and reading about it later.

## What we actually shipped (v1)

Concretely, our first implementation is techniques 0, 1, and 2 — the structural allowlist, regex
patterns, and an entropy sweep — plus the two-detector gate. We deliberately did *not* build the
NER tier (technique 3) or verification-first scanning (technique 4). Here's exactly what's in the
box:

- **Structural allowlist** (the primary control): records are built from named safe fields;
  the only free text kept is two bounded excerpts — the first prompt (capped at 280 characters)
  and a coarse per-tool summary (capped at 200) that records the command's *first word*, a file's
  *basename*, or the literal string `url` — never the payload.
- **Regex named patterns**: a dozen credential patterns, applied specific-first — private-key
  blocks, JWTs, Anthropic / OpenAI / Stripe / GitHub / AWS / Shopify / Slack keys, bearer tokens,
  and `user:pass@host` URL userinfo — plus optional email redaction.
- **Entropy sweep**: any 32-plus-character run whose Shannon entropy clears a threshold, with
  UUIDs and tool-call ids explicitly exempted so our own database keys survive. (This is the
  classic Shannon approach — the one the newer token-efficiency work is starting to replace. We're
  on Shannon in v1 by choice: it's zero-dependency and it sits behind the allowlist and the second
  detector, so it doesn't have to be state-of-the-art to be safe.)
- **Structured-PII backstop**: because credentials aren't the only sensitive thing that can land
  in an excerpt, we also redact phone numbers, US SSNs, credit cards (Luhn-validated, so a random
  16-digit id isn't mistaken for a card), IPv4/IPv6/MAC addresses, `password: …`-style
  assignments, and a partial US-address/ZIP. Every one is paired with a look-alike it must *not*
  eat — a store count, a version number, a short code — so the backstop can't quietly corrupt the
  metrics-safe text. What it deliberately doesn't attempt is *unstructured* PII: names and
  free-form addresses need the NER tier, and (per the benchmark above) bolting regex onto that
  buys almost nothing. That's a documented gap, not a silent one.

And it's tested, not asserted: correctness tests cover the load-bearing cases — no planted
secret survives a line-by-line pass, a bare password inside a `postgres://user:pass@host` URL is
caught, every PII category is redacted while its look-alike survives, UUIDs and tool-ids survive,
ordinary prose is left untouched, and extraction never stores a raw command body.

The one structural fix worth calling out is what we *stopped* shipping. We briefly synced a
scrubbed copy of the whole transcript between machines — which is exactly the "lean on regex over
arbitrary text" trap this post warns about. We replaced it with an allowlist-built *event log*
(what tools ran, what agents did, the metrics) built from the same safe fields as the derived
record. That change did more for PII safety than any pattern we could add: it took the arbitrary
free text off the wire entirely, so the regex backstop is back to guarding two short excerpts
instead of a whole transcript. The strongest move was, again, to collect less.

### What it costs

We profiled it, because "is this fast enough?" deserves a measured answer, not a shrug. The
in-process scrubbing is genuinely cheap; the one expensive step is the second detector, because it
shells out to a separate process:[^bench]

The shape of that chart is the whole argument for how we deploy the second detector. Redacting an
excerpt is around fifty microseconds — even with the added PII patterns, still microseconds; a
single gitleaks invocation is tens of milliseconds — hundreds of times the cost of the in-process
pass, because forking a process dwarfs any regex. That's exactly why gitleaks runs *best-effort*
on the hot paths and is *required* only on the one boundary where the guarantee has to be
airtight — the cross-machine sync. You pay for the expensive detector where it buys the most
safety, and not on every excerpt. (It's also a quiet argument for the event log: a full
transcript scrub is milliseconds of regex over arbitrary text; an event log is a handful of
already-safe fields.)

## The honest ranking

If there's one takeaway, it's that these techniques aren't interchangeable and the ML one isn't
automatically the best. The order that's served us:

- **Structural allowlist** — strongest, because it doesn't detect, it excludes. Use wherever you
  own the schema.
- **Regex named patterns** — precise and free on structured secrets. The workhorse.
- **Entropy / token-efficiency** — the catch-all for novel tokens; move from Shannon entropy toward
  BPE-style signals.
- **NER / zero-shot models** — the only thing that catches unstructured PII, at the cost of
  over-redaction and CPU. Add deliberately.
- **Verification-first scanning** — highest confidence on real credentials; too slow for the edge,
  perfect for a scheduled sweep.

We lead with the boring one on purpose. Every technique after technique zero is a guess about what
counts as sensitive, and guesses have a false-negative rate. The allowlist has none, because it
never guesses — and the cheapest secret to scrub is the one you were never holding.
