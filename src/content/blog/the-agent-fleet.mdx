*New here? [Start with the whole system](/blog/start-here-the-whole-system) — what EarlBear is and the stack this runs on.*

EarlBear's overnight work is done by a **fleet of agents** — codenamed, each with
one job, all running from a single codebase. Two are live and working tonight; two
are freshly scaffolded and coming online. This post is the roster: who they are,
what each one does, and how they hand work to each other. Each agent has (or will
have) its own post; this is the map that ties them together.

## One codebase, many agents

The fleet is not four repositories. It is **one codebase that runs a different
agent depending on the `AGENT` it is told to be.** Each agent is a self-contained
vertical under `agents/<name>/` — its instructions, its secrets, its tools — sitting
on a shared spine that every agent uses unchanged.

An agent's identity is a **markdown character brief** (`agents/<name>/CLAUDE.md`,
written in the second person: "You are Scout…") plus a small JSON manifest
(`agent.json`) that declares which plugins and CLIs it installs. Secrets ride along
**dotenvx-encrypted** in the repo and are decrypted at boot; capabilities come from
a **shared plugin marketplace** every agent draws from. Adding an agent adds a
vertical — the spine does not change.

## The fleet as a pipeline

The four agents are not four silos. They form a connected pipeline around one shared
`scan_requests` queue, routed by an `is_internal` flag: Scout takes the public
storefronts, Medic takes the owned ones.

Medic sits off to the side of that public pipeline: it is **Scout's internal twin**,

remediating them — the one agent allowed to change a live store, and only through
reversible A/B experiments.

queue and findings tables, live storefronts, Drive — and a human reviewer sits over
all of it. Click any use case to see who does it.

The through-line: every agent produces work a human signs off on. Watson never moves
an issue to Done, Envoy never sends, Medic's every change is reversible by
construction. The fleet does the labor; a person keeps the judgment.

Four agents today — two live, two scaffolded. Each entry links to its own post as
that post lands.

**[Watson](/blog/watson-the-overnight-jira-loop)** — the overnight Jira loop.
*Live.* Works the EARL board: picks up assigned or `ai-eligible` issues, routes each
by its column, drafts what it needs, and leaves it for a human. Never marks its own
work done.

**[Scout](/blog/scout-the-storefront-auditor)** — the storefront auditor. *Live.*
Queue-driven: pulls targets from `scan_requests`, walks each live storefront as a
first-time buyer, streams findings to the dashboard tables, and ships a sanitized
transcript to Drive. Audits only — no Jira or Shopify writes.

**Medic** — the store healer. *Scaffold.* Scout's internal, higher-privilege twin:
audits owned stores (`is_internal = true`) and remediates them through reversible
A/B experiments — propose, experiment, promote the winner or roll back, verify. The
only agent that can change a live store. Post to follow when it goes live.

**Envoy** — the outreach drafter. *Scaffold.* Turns Scout's public findings into
qualified leads: discovers the business contact, drafts a compliant value-first
pitch, and files a lead in Watson's Jira pipeline. Drafts and queues; never sends —
a human reviews and sends every message. Post to follow when it goes live.

## How a new agent joins

Adding an agent is a repeatable move, not a rebuild. The `add-agent` skill scaffolds
the folder, the encrypted-secret pair, and the manifest; two more skills wire its
plugins and secrets. Because every agent is just a vertical over the shared spine,
the fleet grew from one agent to four in about a week without the spine changing.

When a new agent lands, it shows up here two ways: it gets its own post in this
series, and — because every fleet post shares the `fleet` tag — it appears
automatically on the [fleet tag page](/tags/fleet/). That page is the always-current
list; this post is the curated tour.
